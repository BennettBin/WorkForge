# WorkForge AI 实施计划（面向 AI 开发者）

## 当前执行决策
- 本轮只执行并验收到 **MVP-1**，对应步骤 **1 到 22**。
- 步骤 23 到 36 暂不启动，待 MVP-1 稳定后再进入下一阶段。
- MVP-1 输入文件范围：`PDF / DOCX / DOC / TXT / PPT / PPTX`，本阶段只实现“生成 PPT”任务闭环。

## 范围与约束
- 目标：先完成 MVP-1（PDF/DOCX 到 PPTX 闭环），再按 MVP-2 到 MVP-6 逐步扩展。
- 技术栈：Tauri 2、React + TypeScript + Vite + Ant Design、FastAPI、JSON/Excel Repository、HTTP + WebSocket。
- 执行原则：每步只做一件事；每步完成后必须先测通再进入下一步。
- 账号体系：MVP-1 必须支持多用户注册、登录、密码修改、用户信息修改。
- 模型通道：MVP-1 必须支持在线 API、Ollama、本地 LLM 三种方式；默认在线通道使用 DeepSeek API。

## MVP-1 关键参数冻结
- 文件大小上限：单文件 `50 MB`。
- 任务文件数：每任务最多 `1` 个源文件。
- 页数上限：单任务最多 `30` 页；默认 `10` 页；推荐输入范围 `5-20` 页。
- 并发策略：同一时间最多运行 `1` 个任务；等待队列最多 `5` 个任务。
- 文件保留：默认保留 `7` 天，支持用户手动删除；后端保留 `created_at / updated_at / expires_at` 字段。
- 兼容目标：PPTX 验收必需目标为 Microsoft PowerPoint 365（Windows）；WPS 作为参考兼容。
- 模板策略：MVP-1 仅提供 `1` 个“学术简洁模板”。
- 日志可见性：用户侧展示阶段与摘要；开发者侧保留完整日志与排障信息。
- 发布策略：MVP-1 验收以开发环境闭环为准；Tauri 安装包打包与安装卸载验证放到发布阶段。

## MVP-1 稳定完成判定标准
- 闭环可用：用户可完成“上传文件→创建任务→执行生成→下载 PPTX”全流程。
- 结果可用：导出的 PPTX 在 Microsoft PowerPoint 365（Windows）可正常打开，且满足“封面页 + 内容页 + 总结页 + 可打开”最低标准。
- 流程稳定：MVP-1 回归集连续运行 3 次通过，无阻断级失败。
- 异常可诊断：上传失败、解析失败、生成失败均有可读错误与可追踪记录。
- 数据一致：任务、文件、输出版本记录一致，无脏数据或丢失版本。
- 解析成功率：`>=95%`，仅统计合法样本（非加密、非损坏、非空、未超限、标准格式）；
  非法或异常样本（加密、损坏、空文件、超限等）需被正确识别并返回清晰错误。

## MVP-1 重试与超时策略
- 上传：不自动重试，超时 60 秒，支持手动重传。
- 解析：自动重试 1 次，超时 60 秒，支持手动重跑。
- 大纲生成：自动重试 2 次，超时 120 秒，支持手动重跑。
- 内容生成：按页执行，每页自动重试 2 次，单页超时 90 秒，支持手动重跑。
- 质量审查：不自动重试，超时 60 秒；审查失败后进入自动修正。
- 自动修正：最多 1 次，超时 120 秒。
- 导出：自动重试 1 次，超时 60 秒，支持手动重跑。
- 下载：不自动重试，超时 60 秒，支持手动重试。
- 总任务超时：15 分钟。
- 手动重跑规则：保留原 `task_id`，追加 `AgentRun`；若重新导出则创建新 `OutputFile` 版本，不覆盖旧版本。

## MVP-1 回归样本要求
- 样本提供：产品负责人提供真实/近真实正常样本；开发者补充异常样本。
- 最小样本集：12 个文件（4 正常 PDF、4 正常 DOCX、1 非法类型、1 空文件、1 损坏或加密、1 超限或模拟超限）。
- 通过阈值：连续运行 3 次，每次全部用例通过；异常样本的“通过”定义为正确拒绝并返回明确错误。

## 分步指令

### 步骤 1：冻结产品范围与验收口径
- 指令：把 `product_design.md` 中 MVP-1 到 MVP-6 的范围和“不做项”整理成一页需求基线文档，并标注优先级。
- 验证测试：产品负责人逐条确认基线文档，确认后无“高优先级需求缺失”。

### 步骤 2：冻结技术基线
- 指令：把 `tec_stack.md` 的推荐主栈、版本下限、通信方式、存储策略整理成技术基线文档。
- 验证测试：技术负责人审核并签字，确认不存在版本冲突和替代方案未决项。

### 步骤 3：建立目录骨架
- 指令：按产品文档建议创建前后端、存储、文档目录，并保留 MVP 所需最小结构。
- 验证测试：目录检查清单 100% 通过，缺失目录数为 0。

### 步骤 4：定义统一任务状态机
- 指令：将正常状态、失败状态、修订状态整理为单一状态机规范，明确允许的状态迁移。
- 验证测试：用 3 条典型流程（成功、失败、修订）走查，均能在状态机中闭环。

### 步骤 5：定义核心数据模型契约
- 指令：基于产品文档固定 Task、File、AgentRun、SkillCall、LLMProviderConfig、OutputFile 的字段与必填约束。
- 验证测试：模型契约评审通过；每个模型至少有 1 个有效样例和 1 个无效样例被正确判定。

### 步骤 6：定义 Repository 抽象接口
- 指令：先定义数据访问接口，不实现数据库细节，确保后续可从 JSON/Excel 平滑迁移。
- 验证测试：接口评审通过；接口已覆盖所有核心模型的创建、查询、更新、版本读取操作。

### 步骤 7：实现 JSON/Excel 存储适配层
- 指令：完成 MVP 阶段的数据落盘与读取策略，包含并发写入保护与异常恢复规则。
- 验证测试：连续执行 50 次任务写入与读取，数据一致性检查通过，且无损坏记录。

### 步骤 8：建立后端基础服务
- 指令：初始化 FastAPI 服务，完成健康检查、配置加载、统一错误响应格式。
- 验证测试：健康检查接口稳定返回成功；错误请求返回统一格式且包含可追踪标识。

### 步骤 9：建立前端基础壳
- 指令：初始化 React + TypeScript + Vite + Ant Design，搭建页面路由与全局状态管理框架。
- 验证测试：前端可正常启动并完成页面跳转；控制台无阻断级错误。

### 步骤 10：建立桌面壳集成
- 指令：接入 Tauri 2，完成前后端本地联动启动与关闭流程。
- 验证测试：桌面应用可一键启动；关闭应用后不残留后端孤儿进程。

### 步骤 11：实现文件上传链路
- 指令：完成 PDF/DOCX/DOC/TXT/PPT/PPTX 文件上传、元数据记录、存储路径管理，并实现 50MB 限制与单任务单文件约束。
- 验证测试：6 类合法文件上传成功；空文件、超限文件、非法类型文件可被正确拒绝并返回可读错误。

### 步骤 12：实现文件解析链路
- 指令：完成 PDF/DOCX/DOC/TXT/PPT/PPTX 解析，生成解析产物并建立与任务关联；对扫描/图片型 PDF 增加 OCR 路径与异常识别。
- 验证测试：合法样本解析成功率达到 >=95%；异常样本均返回清晰、可读、可追踪错误原因。

### 步骤 13：实现任务创建接口
- 指令：实现任务创建、参数校验、初始状态写入、任务与文件绑定。
- 验证测试：有效请求可创建任务；缺失必填字段时返回可读错误且不落脏数据。

### 步骤 14：实现 Task Orchestrator 主流程
- 指令：按“解析完成→规划→生成→审查→导出”建立统一编排流程，并记录阶段事件。
- 验证测试：单任务执行可完成全阶段流转，阶段顺序与状态机规范一致。

### 步骤 15：实现基础多 Agent 分层职责
- 指令：落地 Coordinator、Task Agent、Sub Agent 的职责边界和输入输出协议。
- 验证测试：一次任务中可观测到各 Agent 产出；Task Agent 能阻断不合格子结果进入导出。

### 步骤 16：实现模型路由最小能力
- 指令：建立 Model Router，支持在线 API、Ollama、本地 LLM 三类通道，并默认走 DeepSeek 在线 API。
- 验证测试：三类通道均可完成连通性验证；默认配置生效且策略缺失时有可预期降级逻辑。

### 步骤 17：实现 PPT 大纲生成
- 指令：基于解析内容和用户要求生成页级大纲，包含标题、页目标和重点。
- 验证测试：抽样 10 个任务，大纲页数与用户要求一致，且结构完整度达标。

### 步骤 18：实现 PPT 内容生成
- 指令：按大纲逐页生成正文与备注建议，保留来源追踪信息以便后续修订。
- 验证测试：每页均有内容产出；空页率为 0；来源追踪字段完整。

### 步骤 19：实现质量审查流程
- 指令：按产品文档中的输出标准执行自动审查，未通过项必须进入修正分支。
- 验证测试：构造 3 类缺陷样本（结构、风格、事实一致性），均能被识别并拦截。

### 步骤 20：实现 PPTX 导出能力
- 指令：输出可下载的 PPTX 文件，并记录输出版本元数据。
- 验证测试：导出文件可在目标办公软件正常打开，且页数与审查通过版本一致。

### 步骤 21：打通 MVP-1 前端闭环
- 指令：在前端完成“注册/登录→上传文件→创建任务→查看进度摘要→下载结果→修改用户信息/密码”的最小闭环。
- 验证测试：端到端手工测试通过，用户可在单次会话内完成完整闭环并成功拿到 PPTX。

### 步骤 22：补齐 MVP-1 自动化回归
- 指令：为上传、解析、编排、导出建立最小自动化回归集。
- 验证测试：回归集连续运行 3 次全部通过，无不稳定失败。

### 步骤 23：实现任务进度流（MVP-2）
- 指令：通过 WebSocket 推送阶段进度、当前 Agent、关键日志摘要。

### 步骤 24：实现 Agent 时间线视图（MVP-2）

### 步骤 25：实现 LLM Provider 配置（MVP-3）
- 指令：支持 OpenAI-compatible、Ollama、Custom Base URL 的配置、保存、启停与默认选择。
- 验证测试：三类 Provider 均可完成连通性测试；错误凭证能返回明确失败原因。

### 步骤 26：实现模型配置页面（MVP-3）

### 步骤 27：实现 Skill Registry（MVP-4）

### 步骤 28：实现 Skill 调用审计（MVP-4）
